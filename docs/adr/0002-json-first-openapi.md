# ADR 0002: JSON-First OpenAPI

Status: Accepted

## Context

AKB exposes multiple REST API surfaces:

- the Next.js web/API bridge in `apps/web`,
- Registry API,
- Ingestion Service,
- RAG Retrieval Service,
- LLM Gateway Service,
- Evaluation Service,
- Governance Service,
- Platform Infrastructure status service.

Historically, service-local `openapi.yaml` files and runtime `/openapi.json`
were the canonical service contracts. The central application standard requires
`openapi/openapi.json` as the repository-level JSON-first contract.

## Decision

AKB adds `openapi/openapi.json` as the root JSON-first OpenAPI contract. The
file is generated by `scripts/generate_openapi_index.rb` from service-local
OpenAPI YAML files and the web API route tree.

Service-local `openapi.yaml` files remain in place during migration because
service Dockerfiles and service README files already reference them. They are
legacy source contracts until the service generation workflow is unified.

## Compatibility

No API paths, methods, payloads, or response shapes are changed by this
decision. The root OpenAPI records current endpoints and keeps the existing AKB
error field `trace_id`.

The central standard's `requestId` error field is a planned compatibility-safe
follow-up. It must not be introduced by breaking existing clients.

## Consequences

- CI validates `openapi/openapi.json` as JSON and lints it with Redocly.
- API changes must regenerate the root OpenAPI contract.
- If `openapi.yaml` is later kept as generated output, it must be marked as
  generated from `openapi/openapi.json`.
